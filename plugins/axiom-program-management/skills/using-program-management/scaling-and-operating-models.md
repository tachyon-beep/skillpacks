# Scaling and Operating Models

**An operating model is machinery for solving a coordination problem. Adopt one before you can name the specific coordination problem and you buy ceremony without coordination — a full calendar, trained staff, a wall of new roles, and the same handoffs still failing. Most "we need to scale" is not a scaling problem at all: it is a descope problem, a WIP problem, or a dependency problem wearing a framework's clothes. The discipline here is to diagnose the coordination pain precisely, fix it at the cheapest altitude that resolves it, and reach for a heavier operating model only when a specific, named pain demands a specific, named structure. Scale value-flow first; scale ceremony last.**

This sheet is one altitude above [`cross-project-dependencies-and-integration.md`](cross-project-dependencies-and-integration.md). That sheet owns the *mechanics* of the individual coordination instruments — how big-room/PI planning surfaces the cross-team dependency graph, how a release train aligns handoffs onto a shared boundary, how a scrum-of-scrums runs the day-to-day. This sheet is about the *whole operating model*: whether to adopt one of the named frameworks at all, which one fits which coordination problem, and how to shape the team boundaries so the coordination problem shrinks before you spend any machinery on it. Read the other sheet for how a release train works; read this one for whether you should be running a train.

## When to scale — and when not

The instinct under growth pressure is to add coordination machinery: more synchronization meetings, a scaling framework, a program management office, a layer of coordinators. This is almost always the wrong first move, because the felt symptom — "coordination is getting harder, we need to scale" — is rarely a coordination-capacity problem at its root. Three cheaper diagnoses come first, and each is a different sheet:

- **It is a WIP problem.** The teams feel slow and tangled because everything is started and nothing is finishing; cross-team work piles up at the seams because every team is at 100% utilization and nothing can be received. Capping WIP ([`delivery-cadence-and-flow.md`](delivery-cadence-and-flow.md)) often dissolves the apparent coordination crisis, because most of the "coordination" was teams chasing each other for work that was blocked, not work that needed a framework to sequence.
- **It is a scope problem.** The program is trying to do too much at once across too many teams, so everything depends on everything. Descoping to a thinner slice ([`scope-and-backlog-management.md`](scope-and-backlog-management.md)) cuts dependencies faster than any synchronization layer manages them. Fewer things in flight means fewer seams to coordinate.
- **It is a dependency problem solvable with a contract, not a framework.** Two teams keep surprising each other; the fix is two dated, owned dependency contracts and a weekly sync ([`dependencies-and-coordination.md`](dependencies-and-coordination.md)), not a Program Increment cadence across nine teams. A framework is a sledgehammer; some coordination problems are two-nail problems.

Adding coordination machinery is a last resort because it has a permanent cost. Ceremony, once installed, ossifies: the planning event that solved a real problem keeps running on the calendar after the problem is gone, the coordinator role outlives the dependency it was created to own, and the framework becomes the thing you serve instead of the thing that serves you. Cheap fixes (cap WIP, descope, contract a seam) are reversible; a framework rollout is not. So diagnose first, and scale only the part of the machinery that the named, residual coordination pain actually requires.

## Diagnose the actual coordination problem first

A framework solves a coordination problem. The corollary, routinely ignored, is that you cannot pick the right framework — or know whether you need one — until you have named the specific problem in operational terms. "We need to scale" is not a problem statement; it is an anxiety. The problem statement is a sentence of the form *"these specific handoffs fail / these specific decisions stall / these specific dependencies surprise us, and here is the mechanism by which they do."*

Force the diagnosis to that level of specificity before considering any model:

- **Which handoffs fail, and why?** Is it that handoff dates are never agreed (a contract problem), or that teams march to different sprint calendars so handoffs land at arbitrary points (a cadence-alignment problem), or that the work is sliced so a feature can't be delivered without three teams moving in lockstep (a boundary problem)?
- **Which decisions stall, and where?** Is it that no one has the authority to arbitrate cross-team priority (a decision-rights problem — [`program-structure-and-governance.md`](program-structure-and-governance.md)), or that decisions are made but never propagate (a communication problem)?
- **Which dependencies surprise, and at what distance?** Are they pairwise seams a board would catch, or transitive multi-hop seams that no single team has line of sight to (the case that genuinely needs a graph owner)?

Each of these maps to a *different* part of an operating model — a cadence, a decision forum, a synchronization instrument, a redrawn boundary. Naming the mechanism tells you which part you need, and — just as often — that you need one part, not a whole framework. A team that adopts the entire SAFe calendar to fix "handoff dates are never agreed" has installed Program Increment planning, release trains, and a lean-budgeting layer to solve a problem two dependency contracts would have closed.

## Lean, neutral reads of the named frameworks

The three frameworks below are not religions and not strawmen. Each is a coherent answer to a real coordination problem, with real strengths and real costs. The discipline is to read each by *what coordination problem it solves and what it charges to solve it*, then pick by problem — not by what is fashionable, not by what the last organization used, and not by which one a consultant is selling. A fair test of your own read: would an advocate of each framework recognize the description as honest? If your "what it offers" column is thin and your "what it costs" column is fat, you have written a strawman, not a diagnosis.

| Framework | What it offers | What it costs | When it fits |
|-----------|----------------|---------------|--------------|
| **SAFe** (Scaled Agile Framework) | A full, prescriptive operating model for many teams: **Program Increment (PI) planning** to surface and contract the whole cross-team dependency graph in one synchronized event, **Agile Release Trains** to align teams on a shared cadence and integration rhythm, lean budgeting that funds value streams rather than projects, and explicit roles for cross-team coordination. It is the most complete out-of-the-box answer when the coordination problem is large and you need structure now. | Ceremony-heavy and batch-oriented: the PI cadence is a multi-sprint batch boundary with the batch costs that implies; the role and event inventory is large and can ossify into process-for-its-own-sake; the prescription can crowd out the team-level adaptation that made the underlying agile practices work. Easy to adopt the ceremonies and miss the intent. | Large programs (many teams, real cross-team dependency density) that need a coherent coordination structure quickly and lack the time to evolve one; contexts where a shared, named vocabulary across many teams is itself valuable; regulated or deadline-bound programs that benefit from the predictive cadence. |
| **LeSS** (Large-Scale Scrum) | Scaling Scrum with the **minimum** added process: *one* product backlog and *one* product owner across all teams, **feature teams** that each can deliver a whole customer-centric feature (so most dependencies become intra-team), and the same Scrum events run across teams with as little new machinery as possible. The philosophy is descaling-by-design — shrink the coordination problem rather than build apparatus to manage it. | Demands genuinely cross-functional feature teams and a single empowered product owner — organizationally hard; offers little prescriptive scaffolding, so an organization that wanted a detailed playbook gets principles instead and must do the design work itself; strains past a few teams sharing one backlog. | Organizations that can form true feature teams and want to minimize added process; product-centric work where one backlog and one product owner are realistic; teams that prefer to redraw boundaries over adding coordination layers. |
| **Scrum@Scale** | A **modular** scaling model: a scaled scrum-of-scrums (a "Scrum of Scrums") for cross-team delivery coordination and a parallel Product Owner cycle for cross-team prioritization, composed only as far as the org actually needs. You adopt the modules the coordination problem requires and leave the rest — explicitly designed for incremental, à la carte scaling. | The modularity that is its strength is also its demand: it gives a frame, not a filled-in playbook, so the organization must decide which modules it needs and how they connect; weak facilitation lets the scaled forums degrade into status meetings. | Organizations that want to scale incrementally and compose only the coordination they need; situations where SAFe is too prescriptive and LeSS's single-backlog constraint is unrealistic; growing-into-scale contexts where the coordination problem is still revealing itself. |

The honest cross-framework read: SAFe gives you the most structure and charges the most ceremony; LeSS gives you the least added process and demands the most organizational change to earn it; Scrum@Scale sits between, trading completeness for composability. None is "the agile way to scale." Each is a bet about which is cheaper for *your* coordination problem — added ceremony, organizational restructuring, or self-assembly. Pick by which cost you can afford against which problem you actually have.

## The governing principle: fit the model to the problem, scale ceremony last

The single rule that subsumes the rest: **fit the operating model to the coordination problem, and scale value-flow before you scale ceremony.** Value-flow is the work moving from idea to realized outcome — WIP discipline, slicing, dependency contracts, the cadence that lands handoffs cleanly. Ceremony is the meeting and role structure layered on top to coordinate that flow. The default failure of scaling is to install the ceremony (the events, the roles, the framework) while the value-flow underneath is still broken — unbounded WIP, oversized slices, uncontracted seams — so the new ceremonies coordinate a flow that was never going to be predictable regardless of how well coordinated it was.

Scale value-flow first means: get WIP under control, slice work so features fit inside team boundaries, make dependencies dated owned contracts, and align the cadence — *then* see how much residual coordination problem remains, and add exactly the ceremony that residual demands. Most of the time, fixing the flow shrinks the coordination problem enough that the heavy framework you were about to adopt is no longer needed. The ceremony you add last is the ceremony you actually need; the ceremony you add first is the ceremony you will be unable to remove later.

## Team topologies: shaping the dependency graph at the source

The most powerful scaling lever is not a coordination layer over the teams — it is the *team boundaries themselves*. **Conway's law** observes that a system's architecture comes to mirror the communication structure of the organization that builds it. Read as a lever rather than a curse (the "inverse Conway maneuver"): *choose the team boundaries to shape the dependency graph you want.* If you draw boundaries so each team owns a whole vertical slice of value, most dependencies become intra-team and never reach the coordination layer at all. If you draw them so every feature crosses four teams, you have manufactured a coordination problem that no framework will fully tame.

The Team Topologies model gives four team types and three interaction modes. The point is not the vocabulary; it is that team type and interaction mode are *design choices* you make to minimize each team's cognitive load and minimize inter-team dependencies.

| Team type | Purpose | Reduces coordination by |
|-----------|---------|-------------------------|
| **Stream-aligned** | Owns a whole slice of value end-to-end (a product area, a user journey — e.g. a checkout flow or a reporting feature) and delivers it without handing off. The default team type; most teams should be this. | Keeping the dependencies of a value stream *inside one team* — the feature-team idea. A handoff that would cross teams becomes work inside one team. |
| **Platform** | Provides internal services (deployment, data, shared infrastructure) that stream-aligned teams self-serve, ideally as a product with a clean interface. | Turning a recurring cross-team dependency into a self-service capability — the consumer pulls from a stable interface instead of negotiating a handoff each time. |
| **Enabling** | A specialist team that coaches stream-aligned teams to acquire a missing capability, then steps back. Temporary by design. | Removing a *standing* dependency on scarce expertise by transferring the capability into the stream-aligned team, rather than routing every request through a bottleneck. |
| **Complicated-subsystem** | Owns a part requiring deep specialist knowledge (a pricing engine, a physics core) that is impractical to spread across stream-aligned teams. | Encapsulating irreducible complexity behind one team and one interface, so other teams depend on the interface, not the complexity. |

The three **interaction modes** name how two teams should relate: **collaboration** (two teams work closely for a bounded period to figure out a new interface — high-bandwidth but expensive, deliberately temporary), **X-as-a-service** (one team consumes another's capability through a stable interface with minimal ongoing coordination — the low-cost steady state you want most dependencies to settle into), and **facilitating** (an enabling team helps another team get unblocked, then withdraws). The discipline is to drive dependencies *toward* X-as-a-service — a clean self-service interface — and to treat collaboration as a temporary, costly mode you exit once the interface is understood, not a permanent way of working.

## Descaling is a real option

The move the frameworks under-emphasize: **fewer dependencies beats more coordination machinery.** Before adding a synchronization layer to manage a cross-team dependency, ask whether you can *redraw the boundary so the dependency becomes intra-team.* This is descaling — the LeSS philosophy and the stream-aligned-team idea, applied as a first response rather than a last one.

The mechanism is direct. A dependency between Team A and Team B costs a contract, a handoff date, a synchronization point, and the risk that the handoff slips. Move the work so one team owns both sides and that entire coordination cost vanishes — the dependency is now a within-team task, coordinated by the team's own standup at zero marginal machinery. Re-drawing a boundary so a recurring cross-team seam becomes one team's internal work is almost always cheaper than building and operating the apparatus to coordinate that seam forever. A program with a chronic, high-traffic seam between two teams should first ask whether those two teams should be one team, or whether the slice should be re-cut so the feature lives inside a single stream-aligned team — before it builds a release train to coordinate the handoff. The lever is Conway's law again: change the org boundary to change the dependency graph. Coordination machinery manages dependencies; descaling *removes* them, and removal dominates management.

## The migration path: evolutionary, not big-bang

Operating models should be grown, not installed. The failure mode is the big-bang framework rollout: pick SAFe (or any framework), train everyone, restructure the org, fill the calendar, and flip the switch on a Monday — at which point the organization is running a coordination machine sized for problems it may not have, while the problems it *does* have are buried under the new ceremony. Evolutionary adoption inverts this: **start minimal, and add a specific structure only when a specific named pain demands it.**

Concretely: start with stable teams and the cheapest coordination that works — dated dependency contracts and a weekly cross-team sync. When a specific pain emerges and is named — "we keep discovering cross-team dependencies at integration because we plan in isolation" — add the one instrument that addresses *that* pain (synchronized big-room planning), and only that. When the next named pain emerges — "handoffs land at arbitrary points because our cadences differ" — add a shared cadence (a release train). Each structure earns its place by closing a named pain, which means you can also tell when it has stopped earning its place and remove it. You may converge on something that looks like one of the named frameworks — that is fine. The difference is that you arrived there by accreting structures that each solved a real problem, so every piece is load-bearing, rather than by importing a whole framework and spending years discovering which two-thirds of it your context never needed.

## The honest close: where heavier structure is genuinely warranted

Lean is the default, not a religion — and the honest discipline is to recognize where it stops being right. In **regulated, safety-critical, or very large programs**, heavier predictive structure is not bureaucratic overhead; it is correct. A program delivering against a medical-device certification, an avionics standard, or a contractual regulatory deadline genuinely needs **formal stage gates, detailed up-front plans, fixed-scope commitments, and auditable traceability** — because the cost of a late-discovered defect or a missed compliance obligation is catastrophic and irreversible, and because external authorities require evidence that the process was followed, not just that the outcome shipped. Treating that structure as anti-agile bureaucracy to be stripped out is exactly the dogmatism this pack rejects. The predictive plan is the right tool when the variance you are managing is unacceptable rather than merely inconvenient.

This pack runs the *operational delivery* even in those contexts — the working RAID log, the honest status, the governance cadence that meets and decides. But the **formal process maturity** that regulated and safety-critical contexts demand is a different discipline, and it routes elsewhere. **For CMMI maturity levels, formal stage gates, the requirements lifecycle and traceability matrix, formal Decision Analysis and Risk Management process areas, and statistical process control, hand off to `/axiom-sdlc-engineering`.** The rule of thumb, mirrored from the router: **`/axiom-sdlc-engineering` DEFINES the process; this pack RUNS delivery inside it.** A regulated program uses both — that pack for the formal traceability and governance procedures the authority requires, this pack for the day-to-day delivery management that executes them. When the context demands formal maturity, do not improvise it here; route there, then operate inside what it defines.

## Anti-Patterns

1. **Adopting a scaling framework before understanding the coordination problem.** Rolling out SAFe/LeSS/Scrum@Scale wholesale — the full PI cadence, release trains, the role inventory, the whole calendar — at a problem that needed two dependency contracts and a weekly sync. The framework coordinates nothing real because the real coordination need was never named; the organization ends up serving the ceremony instead of being served by it. This is the pack's central scaling failure. *Fix: write the coordination problem as a specific operational sentence — which handoffs fail, which decisions stall, which dependencies surprise — before naming any model, and adopt only the structure that named pain requires.*

2. **Scaling ceremony before scaling value.** Installing the meeting-and-role structure on top of a value-flow that is still broken — unbounded WIP, oversized slices, uncontracted seams — so the new ceremonies coordinate a flow that was never going to be predictable. *Fix: fix the flow first (cap WIP, slice to fit team boundaries, contract the seams, align the cadence); add ceremony only to the residual coordination problem that remains after the flow is sound.*

3. **Copying another organization's operating model wholesale.** Importing the framework, cadence, and role structure that worked somewhere else because it worked there — without checking whether the coordination problem, team boundaries, and constraints are the same. An operating model is an answer to a specific problem; the answer does not transfer just because the question rhymes. *Fix: diagnose your own coordination problem from scratch and let the model follow the diagnosis; borrow patterns, never the whole machine.*

4. **Adding coordination machinery instead of removing dependencies.** Reaching for a synchronization layer to manage a chronic cross-team seam when the cheaper move is to redraw the boundary so the dependency becomes one team's internal work. Machinery manages dependencies forever; descaling removes them once. *Fix: before building apparatus to coordinate a recurring seam, ask whether the boundary can be re-cut — one stream-aligned team owning both sides — so the seam disappears (Conway's law as the lever).*

5. **One-size-fits-all framework dogma.** Treating a single operating model — whether a named scaling framework or "pure lean/kanban everywhere" — as universally correct, and either forcing a heavyweight framework onto a two-team problem or stripping necessary predictive structure out of a regulated, safety-critical program because it "isn't agile." *Fix: hold the model loosely; match rigor to scale and stakes; concede in good faith that large/regulated programs legitimately need formal gates and fixed-scope plans, and route that formal maturity to `/axiom-sdlc-engineering`.*
